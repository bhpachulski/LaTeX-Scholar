\section{Definição de Escopo}

\begin{itemize}

\item PMBOK

    \begin{itemize}
      \item Iniciação: processo que reconhece formalmente que um novo projeto existe. Liga o projeto aos demais trabalhos realizados pela organização e tem como objetivos principais a autorização e obtenção do comprometimento da organização para a formalização de um novo projeto. Não deve ser incluída porque a maioria das informações que devem constar no documento de resultados deste processo foram considerados na etapa de Escolha de Tema. 
      \item Planejamento do escopo: desenvolvimento de uma declaração de resultados principais do projeto. Estão inclusos os critérios que avaliarão se o projeto foi completado com sucesso. Determina os limites do trabalho. (Os itens do template podem ser aqueles da pg.60, livro Possi, P05.2.5)
      \item Definição do escopo: subdividir as principais entregas (deliverables) do projeto, definidas no planejamento do escopo em componentes menores. O principal produto é a EAP. 
      \item Verificação do escopo: Processo de formalização do aceite do escopo do projeto por todos os envolvidos. Requer uma revisão dos produtos de trabalho e dos resultados, de modo a garantir que tudo foi completado satisfatoriamente. Em ambiente acadêmico geralmente é realizada em exame de qualificação. Há um documento da instituição que registra o aceite do escopo.   
      \item Controle de alterações do escopo: Concentra na avaliação dos fatores que criam mudanças no escopo. 
    \end{itemize}
         
\item CMMI (Project Management, Project Planning, pg 194)

		\begin{itemize}
		  \item Desenvolver EAP: A EAP fornece uma esquema para organizar o projeto em relação aos produtos que devem ser fornecidos. 
		  \item Identificar os pacotes de trabalho em detalhes de forma a permitir estimar tarefas, responsabilidades e cronograma. 
		  \item Identificar produtos de trabalho que serão adquiridos externamente. 
		  \item Identificar produtos de trabalho que serão reusados.
		\end{itemize}

\item ISO/IEC 12207 (Management process, pg 43)
  
     \begin{itemize}
        \item O processo de gerenciamento deve ser iniciado estabelecendo-se os requisitos do processo a ser cumprido.
        \item o gerente deve estabelecer a praticabilidade do processo checando se os recursos requeridos para executar e gerenciar o processo estão disponíveis.        
        \item Modificar os requisitos do processo se necessário
     \end{itemize}

\end{itemize}

\section{Postmortem}

\begin{itemize}

\item Collier et al \cite{col96}

\begin{itemize}
      \item Projetar e (promulgate - ?) um survey: elaborar um mecanismo para coletar informações de projeto.
      \item Coletar informações de projeto objetivas: coletar informações que revelem dados importantes do projeto, tais como, custos, disponibilidade de recursos, prazos, deadlines, etc.
      \item Promover reunião com os membros: reunir os membros de forma a coletar informações adicionais, que não foram fornecidas com a realização do survey. 
      \item Promover o ``dia da história do projeto'': reunir um grupo de participantes do projeto com o objetivo de revisar eventos e dados do projeto e usar métodos de gerenciamento de qualidade para tornar explícito elementos chave da dinâmica do processo. 
      \item Publicar os resultados: registrar os resultados, enfatizando lições aprendidas com o objetivo de guiar a melhoria da organização.
\end{itemize}


\item Birk et al \cite{bir02}

\begin{itemize}
\item Preparação: determinar um objetivo para a PMA
\item Coleta de dados: entrevistas semi-estruturadas, discussões de grupo, sessões KJ (post it notes), 
\item Análise: prioriza os tópicos mais importantes. Gera diagrama ishikawa em um processo colaborativo, para identificar causas para experiências positivas e negativas. 
\item Resultados e experiências:  gera um relatório 
\end{itemize}
 
\end{itemize}
